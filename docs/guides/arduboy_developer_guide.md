# Arduboy Emulator — Developer Guide

> Architecture, internals, and how to extend the Arduboy emulator mode.

---

## Architecture Overview

The emulator is a Bus Pirate protocol mode (`src/mode/arduboy/`) consisting of four layers:

```
┌─────────────────────────────────────────────────────────────┐
│                     arduboy.c                               │
│         Mode glue: setup, cleanup, game loop,               │
│         memory management, mode commands                    │
├──────────────┬──────────────────┬───────────────────────────┤
│  avr_cpu.c   │ arduboy_periph.c │    arduboy_term.c         │
│  Interpreter │ SSD1306, GPIO,   │    ANSI renderer,         │
│  (pure CPU)  │ Timer, EEPROM,   │    VT100 key input,       │
│              │ SPI hooks        │    dirty-cell tracking     │
├──────────────┴──────────────────┴───────────────────────────┤
│                  Bus Pirate firmware                         │
│    mem.h (BIG_BUFFER) │ usb_rx/tx │ ui_term │ vt100_keys    │
└─────────────────────────────────────────────────────────────┘
```

### Source Files

| File | Lines | Purpose |
|------|-------|---------|
| `avr_cpu.h` | ~206 | CPU state struct, memory map defines, I/O hook typedefs, inline helpers |
| `avr_cpu.c` | ~1067 | Full AVR instruction decoder, Intel HEX loader |
| `arduboy_periph.h` | ~158 | SSD1306, buttons, timer, EEPROM state structs |
| `arduboy_periph.c` | ~427 | I/O read/write hooks, SSD1306 command processing, button mapping |
| `arduboy_term.h` | ~99 | Terminal renderer and input declarations, palette struct |

| `arduboy.h` | ~24 | Mode interface declarations |
| `arduboy.c` | ~400 | Mode lifecycle, game loop, memory carve-up, mode commands |

## Memory Layout

The emulator allocates the firmware's shared `BIG_BUFFER` (128KB) and carves it into regions:

```
BIG_BUFFER (128KB = 131,072 bytes)
┌────────────────────────────────────────┐ offset 0
│  AVR Flash (program memory)            │
│  32,768 bytes (32KB)                   │
├────────────────────────────────────────┤ offset 32768
│  AVR Data Space                        │
│  2,816 bytes (regs + I/O + SRAM)       │
│    [0x0000–0x001F] 32 GP registers     │
│    [0x0020–0x005F] 64 I/O registers    │
│    [0x0060–0x00FF] 160 ext I/O regs    │
│    [0x0100–0x0AFF] 2560 bytes SRAM     │
├────────────────────────────────────────┤ offset 35584
│  AVR EEPROM                            │
│  1,024 bytes (1KB)                     │
├────────────────────────────────────────┤ offset 36608
│  Scratch area                          │
│  ~94KB — hex file loading buffer       │
└────────────────────────────────────────┘ offset 131072
```

Allocation: `mem_alloc(BIG_BUFFER_SIZE, BP_BIG_BUFFER_ARDUBOY)` in `arduboy_setup_exec()`.  
Deallocation: `mem_free(arena)` in `arduboy_cleanup()`.

> **Important:** Only one `BIG_BUFFER` consumer can be active at a time. The logic analyzer, scope, and Arduboy emulator are mutually exclusive.

## AVR CPU Interpreter (`avr_cpu.c`)

### Design Philosophy

- **Instruction-level**, not cycle-accurate — one call to `avr_cpu_step()` executes one complete instruction and returns the cycle count
- **No interrupt controller** — Timer0 overflow is handled outside the CPU via `arduboy_periph_tick()`
- **Portable** — no RP2040-specific code, could run on any platform
- **I/O hooks** — all register access in the 0x20–0xFF range is intercepted via function pointers

### Instruction Set Coverage

The interpreter covers all instructions commonly generated by avr-gcc for ATmega32U4:

| Category | Instructions | Notes |
|----------|-------------|-------|
| **Arithmetic** | ADD, ADC, SUB, SBC, SUBI, SBCI, INC, DEC, NEG, COM, ADIW, SBIW | Full SREG updates |
| **Logic** | AND, ANDI, OR, ORI, EOR | |
| **Compare** | CP, CPC, CPI, CPSE | |
| **Branch** | RJMP, RCALL, RET, RETI, BRBS, BRBC (all conditional branches), IJMP, ICALL, JMP, CALL | |
| **Load/Store** | LD/ST with X/Y/Z (all addressing modes: plain, post-inc, pre-dec, displacement), LDS, STS, LDI, MOV, MOVW | |
| **Program Memory** | LPM (Z, Z+), SPM (stub) | |
| **Stack** | PUSH, POP | |
| **Bit** | SBI, CBI, SBIC, SBIS, SBRC, SBRS, BST, BLD, LSR, ASR, ROR, SWAP | |
| **Multiply** | MUL, MULS, MULSU, FMUL, FMULS, FMULSU | |
| **System** | NOP, SLEEP, BREAK, IN, OUT | |

### Adding a New Instruction

If you encounter an unimplemented opcode (the interpreter prints a warning and halts), add it to the `avr_cpu_step()` function:

1. Identify the opcode encoding from the [AVR Instruction Set Manual](https://ww1.microchip.com/downloads/en/DeviceDoc/AVR-InstructionSet-Manual-DS40002198.pdf)
2. Find the correct decode branch in `avr_cpu_step()` — instructions are grouped by their upper nibble
3. Implement the operation, update SREG flags as documented
4. Return the cycle count

Example pattern:
```c
/* Inside avr_cpu_step(), in the appropriate decode branch */
case 0x9408: /* SEC — set carry flag */
    avr_sreg_set(cpu, SREG_C, true);
    return 1;
```

### Intel HEX Loader

`avr_load_hex()` parses standard Intel HEX format (`:LLAAAATT[DD...]CC`):
- Record type 0x00: Data — written to flash at the specified address
- Record type 0x01: EOF — parsing stops
- Other record types are ignored (extended address records are not needed for 32KB flash)

The flash array is pre-filled with `0xFF` before loading (matching erased flash behavior).

## Peripheral Emulation (`arduboy_periph.c`)

### I/O Hook Architecture

The CPU's `avr_data_read()` / `avr_data_write()` inline functions check if the address falls in the I/O range (0x20–0xFF) and call the registered hooks:

```c
// In avr_cpu.h (inline)
static inline uint8_t avr_data_read(avr_cpu_t* cpu, uint16_t addr) {
    if (addr >= AVR_IO_BASE && addr < AVR_SRAM_START && cpu->io_read) {
        return cpu->io_read(addr, cpu->io_ctx);
    }
    if (addr <= AVR_RAMEND) return cpu->sram[addr];
    return 0;
}
```

The peripheral layer registers `io_read_hook()` and `io_write_hook()` during `arduboy_periph_init()`. These are giant switch statements dispatching on the register address.

### SSD1306 OLED Emulation

The SSD1306 is connected via SPI on the real Arduboy. In the emulator:

1. AVR code writes to `SPDR` (SPI data register, addr 0x4E)
2. The I/O write hook detects this and calls `ssd1306_spi_byte()`
3. The D/C pin state (tracked from PORTD bit 4 writes) determines if the byte is a **command** or **data**
4. Commands configure addressing (column range, page range, display on/off)
5. Data bytes are written to the 1024-byte framebuffer at the current position
6. The `framebuffer_dirty` flag is set, which the game loop polls

**Addressing mode:** Horizontal addressing (Arduboy's default). After each data byte, the column advances. When it passes `col_end`, it wraps to `col_start` and the page advances.

**SPI is instant** — no transfer delay, no interrupt. Writing to SPDR immediately processes the byte and sets the transfer-complete flag. This is a deliberate simplification; Arduboy games always poll SPSR.SPIF before the next transfer, so the timing doesn't matter.

### Button Emulation

Arduboy buttons are mapped to ATmega32U4 GPIO pins with internal pull-ups (active-low):

| Button | Port | Pin | Register | Bit |
|--------|------|-----|----------|-----|
| UP | F | PF7 | PINF | 7 |
| DOWN | F | PF4 | PINF | 4 |
| LEFT | F | PF5 | PINF | 5 |
| RIGHT | F | PF6 | PINF | 6 |
| A | E | PE6 | PINE | 6 |
| B | B | PB4 | PINB | 4 |

When the AVR reads a PINx register, the I/O hook returns 0xFF (all high) with the bits for pressed buttons cleared to 0. The `button_state` bitmask is set each frame by the terminal input backend.

### Timer0

Timer0 provides the Arduboy's `millis()` / `micros()` timebase:

- `arduboy_periph_tick()` is called after each CPU step with the cycle count
- Cycles are accumulated and converted to microseconds (at 16MHz: 1 cycle = 62.5ns)
- Every 1000µs, the `millis` counter increments
- TCNT0 is advanced but Timer0 interrupts are not vectored into the CPU

This is sufficient for most games that poll `millis()` for timing, but games that depend on Timer0 Compare Match interrupts for precise timing will need the interrupt controller implemented.

### EEPROM

The EEPROM controller emulates the ATmega32U4's register interface:
- **EEARL/EEARH** (0x41/0x42): Address registers
- **EEDR** (0x40): Data register
- **EECR** (0x3F): Control register — write with EEMPE+EEPE triggers write, EERE triggers read

EEPROM data persists within a single mode session (while Arduboy mode is active) but is **lost when exiting the mode**, since it lives in the `BIG_BUFFER` allocation.

### Adding a New Peripheral

To emulate a new ATmega32U4 peripheral:

1. Add state fields to `arduboy_periph_t` in `arduboy_periph.h`
2. Add register address `#define`s to `avr_cpu.h` (if not already there)
3. Add `case` entries in `io_read_hook()` and `io_write_hook()` in `arduboy_periph.c`
4. Initialize state in `arduboy_periph_init()` / `arduboy_periph_reset()`
5. If the peripheral needs time-based updates, add logic to `arduboy_periph_tick()`

## Terminal Backend (`arduboy_term.c`)

### Display Rendering

The 128×64 monochrome framebuffer is rendered as a 128×32 character grid using Unicode half-block characters:

| Top Pixel | Bottom Pixel | Character | Technique |
|-----------|-------------|-----------|-----------|
| ON | ON | █ (U+2588) | Foreground color only |
| ON | OFF | ▀ (U+2580) | Foreground = ON, Background = OFF |
| OFF | ON | ▄ (U+2584) | Foreground = ON, Background = OFF |
| OFF | OFF | (space) | Background color only |

Colors are set with 24-bit truecolor ANSI sequences:
```
\x1b[38;2;R;G;Bm   — set foreground
\x1b[48;2;R;G;Bm   — set background
```

### Dirty-Cell Tracking

The renderer maintains a copy of the previous frame (`prev_fb`). On each render call:
1. For each character cell, extract the top and bottom pixel from both current and previous framebuffers
2. If both pixels match the previous frame, skip the cell
3. Otherwise, position the cursor and draw the cell

This means static screens (title cards, menus) render essentially for free after the first frame. Only changed pixels generate terminal output.

### Input Pipeline

```
USB serial bytes → rx_fifo_try_get()
    → vt100_key_unget() → vt100_key_read()
        → arduboy_term_map_key()
            → button bitmask (OR'd together)
```

The input loop drains all available bytes each frame, feeding them through the VT100 key decoder to handle multi-byte escape sequences (arrow keys generate `\x1b[A`, `\x1b[B`, etc.). All decoded keys in a single frame are OR'd together, allowing simultaneous button presses.

**Exit detection:** If `map_key()` returns `0xFF` (ESC or Ctrl+Q), `poll_buttons()` immediately returns `0xFF`, which the game loop checks to break.

### Adding a New Palette

Add a `const arduboy_palette_t` in `arduboy_term.c`:

```c
const arduboy_palette_t PALETTE_PINK = { 255, 100, 200,  30, 5, 15 };
```

And declare it `extern` in `arduboy_term.h`. The six fields are: `fg_r, fg_g, fg_b, bg_r, bg_g, bg_b` (0–255 each).

### Adding a New Key Binding

Add a `case` in `arduboy_term_map_key()` in `arduboy_term.c`:

```c
case 'a': case 'A':  return ARDUBOY_BTN_A;  /* WASD alternative */
```

## Game Loop (`arduboy.c`)

The main emulation loop in `game_loop()`:

```
1. toolbar_draw_prepare()        — pause BP toolbar rendering
2. Enter alternate screen buffer  — \x1b[?1049h
3. arduboy_term_init()           — draw border, help text
4. Loop at 60fps:
   a. arduboy_term_poll_buttons() → check for exit
   b. arduboy_periph_set_buttons() → update GPIO state
   c. avr_cpu_step() × N         → run ~266,666 cycles
   d. arduboy_periph_tick()       → advance timers per step
   e. arduboy_term_render()       → ANSI art (if framebuffer dirty)
   f. arduboy_term_status()       — FPS display
   g. sleep_us(remainder)         — frame pacing to 16,667µs
5. arduboy_term_cleanup()        — restore cursor, leave alt screen
6. toolbar_draw_release()        — resume BP toolbar
```

### Frame Timing

At 16MHz emulated clock / 60fps = 266,666 cycles per frame. The loop calls `avr_cpu_step()` one instruction at a time (not batched) so that `arduboy_periph_tick()` can track each instruction's cycle count for accurate timer advancement.

After all cycles are consumed, `time_us_64()` measures wall-clock time and `sleep_us()` pads the remainder to hit the 16,667µs target. If the frame took longer than that (CPU-bound), no sleep occurs and FPS drops below 60.

### Mode Lifecycle

```
User selects ARDUBOY mode
    → arduboy_setup()        — prompt for filename
    → arduboy_setup_exec()   — allocate BIG_BUFFER, init CPU/peripherals, load .hex
    
User types "play"
    → cmd_play_handler()     — calls game_loop() (blocking)

User types "load"
    → cmd_load_handler()     — prompt for new filename, load into flash

User types "info"
    → cmd_info_handler()     — print status

User exits mode (m → HiZ)
    → arduboy_cleanup()      — free BIG_BUFFER
```

## Build System

All 8 source files are listed unconditionally in `src/CMakeLists.txt` under `buspirate_common`. The `#ifdef BP_USE_ARDUBOY` guard in C code handles conditional compilation — if the flag is not defined in `pirate.h`, the mode simply doesn't appear in the mode list and the compiler optimizes away the dead code.

To **disable** the Arduboy emulator: comment out `#define BP_USE_ARDUBOY` in `src/pirate.h`.

To **build**:
```bash
cmake --build ./build --parallel --target bus_pirate5_rev10
```

The emulator adds approximately **4–6KB** to the flash image (exact size depends on compiler optimization).

## Extending the Emulator

### Planned Features (Roadmap)

These are the major remaining features, roughly in priority order:

#### 1. LCD Display Backend
Render the Arduboy framebuffer on the Bus Pirate's ST7789 LCD (240×320) instead of (or in addition to) the terminal. The LCD is 320 wide × 240 tall in landscape; at 2× scale the 128×64 image becomes 256×128, fitting perfectly.

**Implementation sketch:**
- New files: `arduboy_lcd.h` / `arduboy_lcd.c`
- Same `render(fb)` API as `arduboy_term_render()`
- Use `lcd_write_rect()` from `pirate/lcd.c` to blit scaled pixels
- Must run on Core 1 (LCD updates happen there) — use inter-core FIFO to signal dirty frames

#### 2. EEPROM Persistence
Save/load the 1KB EEPROM array to a file on flash storage so game save data survives across sessions.

**Implementation sketch:**
- On mode entry: read `<gamename>.sav` into the EEPROM region
- On mode exit (or periodically): write EEPROM region to `<gamename>.sav`
- Use FatFS `f_open` / `f_write` / `f_close`

#### 3. Interrupt Controller
Implement the AVR interrupt vector table so Timer0 (and potentially Timer1/3/4) overflow interrupts fire at the correct point.

**Implementation sketch:**
- In `arduboy_periph_tick()`, when TCNT0 overflows and TOIE0 is set, set a pending interrupt flag
- In `avr_cpu_step()`, before decoding the next instruction, check for pending interrupts when SREG.I is set
- Push PC, clear SREG.I, jump to the interrupt vector address
- RETI restores SREG.I and pops the return address

#### 4. Audio Output
Route the two speaker pins (PC6/Timer3A, PC7/Timer4A) to the Bus Pirate's PWM output or piezo buzzer.

**Implementation sketch:**
- Track Timer3/Timer4 configuration writes in the peripheral layer
- Compute the output frequency from the timer prescaler and compare match value
- Use `pwm_set_freq()` from `pirate/bio.c` to output on a Bus Pirate I/O pin

#### 5. ROM Launcher Menu
A scrollable menu showing all `.hex` files on the flash drive, rendered in the terminal (or on LCD).

**Implementation sketch:**
- Use FatFS `f_opendir()` / `f_readdir()` to enumerate `*.hex` files
- Render a VT100 menu (similar to the mode selection menu)
- Arrow keys navigate, Enter loads and plays

## Testing

### Unit Testing the CPU

The AVR interpreter (`avr_cpu.c/h`) is pure C with no hardware dependencies. To test:

1. Create a test harness that allocates flash/sram/eeprom arrays
2. Write AVR machine code directly into flash (or assemble with `avr-as`)
3. Call `avr_cpu_step()` and verify register/memory/SREG state

Example (testing ADD instruction):
```c
avr_cpu_t cpu;
uint8_t flash[AVR_FLASH_SIZE] = {0xFF};
uint8_t sram[AVR_RAMEND + 1] = {0};
uint8_t eeprom[AVR_EEPROM_SIZE] = {0xFF};

avr_cpu_init(&cpu, flash, sram, eeprom);

// ADD R1, R2  →  opcode 0x0C12 → little-endian: 0x12, 0x0C
flash[0] = 0x12;
flash[1] = 0x0C;

cpu.r[1] = 10;
cpu.r[2] = 20;

uint8_t cycles = avr_cpu_step(&cpu);

assert(cpu.r[1] == 30);    // R1 = 10 + 20
assert(cycles == 1);
assert(cpu.pc == 1);        // Advanced by 1 word
assert(!(cpu.sreg & (1 << SREG_Z)));  // Result not zero
```

### Integration Testing

Flash a real Arduboy game `.hex` and verify:
- Title screen renders
- Button inputs navigate menus
- Gameplay is responsive
- No CPU halt or unimplemented instruction warnings

### Recommended Test Games

| Game | Complexity | Tests |
|------|-----------|-------|
| **Arduboy bounce** | Simple | Basic rendering, button input |
| **Ardubreakout** | Medium | Timing, collision, EEPROM scores |
| **Micro City** | Complex | Heavy CPU usage, complex rendering |
| **Circuit Dude** | Medium | Puzzle logic, EEPROM save data |

## Code Style

Follow the Bus Pirate firmware conventions:

- **120-column limit**, 4-space indent, attached braces (Mozilla clang-format)
- **Namespace prefix**: `avr_`, `arduboy_periph_`, `arduboy_term_` for subsystem separation
- **Static by default** — only functions declared in headers are non-static
- **Inline helpers** in headers for hot-path data access (e.g., `avr_data_read`)
- **Copyright notice** required on new files
- Format with: `clang-format -i src/mode/arduboy/*.c src/mode/arduboy/*.h`
